🏢 The Business Problem

A small business with 25 employees is running an outdated on-premise Exchange Server 2016 for email, a local Active Directory for user management, and a flat network with no segmentation or VPN. Their IT infrastructure is fragile: hardware failures risk total email loss, remote workers cannot securely access internal resources, and onboarding new employees requires hours of manual configuration.

The goal: Migrate the entire company to Microsoft 365 cloud services, reconfigure their network for security, and automate user provisioning — all with zero downtime and zero data loss.

⚙️ Migration Phases

This project was executed in 4 structured phases, following Microsoft's recommended migration framework.

1

2

Performed a hybrid cutover migration from Exchange 2016 to Exchange Online (M365).

3

Reconfigured VLANs, firewall rules, and deployed a site-to-site VPN for remote access.

Automated user provisioning, license assignment, and group membership via PowerShell scripts.

Phase 1: Active Directory & Entra ID Sync

What Was Done

  • Installed and configured Azure AD Connect on the on-premise domain controller.
  • Configured Password Hash Synchronization (PHS) so users can log in to M365 with the same credentials they use on-premise.
  • Validated that all 25 user accounts synced successfully to Entra ID (formerly Azure AD) in the Microsoft 365 Admin Center.
  • Resolved UPN (User Principal Name) suffix mismatches by adding a custom domain to the on-premise AD.

Best Practice: UPN Suffix

Before syncing, always verify that every user's UPN suffix matches the verified domain in M365 (e.g., user@company.com, not user@company.local). Mismatched UPNs cause login failures after migration.

Precaution: Staging Mode First

Always run Azure AD Connect in Staging Mode first. This lets you verify what will sync without actually writing changes to Entra ID. Only disable staging mode after reviewing the sync preview report.

Phase 2: Exchange to M365 Migration

What Was Done

  • Configured a Hybrid Exchange environment to allow coexistence during the transition period.
  • Performed a Cutover Migration using the Exchange Admin Center migration batch wizard.
  • Migrated all mailboxes (email, calendars, contacts) from Exchange Server 2016 to Exchange Online.
  • Updated MX records and Autodiscover CNAME in DNS to point to Microsoft 365.
  • Decommissioned the on-premise Exchange Server after a 2-week validation period.

Best Practice: MX Record TTL

72 hours before the migration, reduce the MX record TTL to 300 seconds (5 minutes). This ensures that when you flip the MX records to M365, DNS propagation happens quickly and email flow switches over within minutes, not days.

Precaution: Backup Before Cutover

Take a full backup of all mailboxes (PST export) before initiating the cutover migration. If the migration batch fails or data corrupts, you can restore from the PST files. Never skip this step.

Best Practice: Coexistence Period

Keep the on-premise Exchange Server running for at least 2 weeks after migration. This gives you a rollback window. Only decommission the server after confirming all email is flowing correctly through M365.

Phase 3: Network Reconfiguration

What Was Done

  • Segmented the flat network into VLANs: Management (VLAN 10), Staff (VLAN 20), Guest (VLAN 30), and Servers (VLAN 40).
  • Configured inter-VLAN routing on the core Layer 3 switch with ACLs restricting Guest VLAN from accessing internal resources.
  • Deployed a Site-to-Site IPsec VPN between the main office and a branch office, ensuring encrypted traffic between both locations.
  • Configured firewall rules to allow only necessary M365 traffic (ports 443, 587, 993) and block all unnecessary inbound connections.
  • Set up DHCP scopes per VLAN with appropriate DNS server settings pointing to M365 and internal DNS.

Best Practice: VLAN Segmentation

Never run a business on a flat network. Segmenting into VLANs isolates threats (e.g., if a guest device gets infected, it cannot reach internal servers). Apply ACLs between VLANs to enforce least-privilege access.

Precaution: Firewall Change Window

Always implement firewall rule changes during a scheduled maintenance window (e.g., after business hours). Test connectivity to all M365 services immediately after applying new rules. Keep a rollback plan ready in case rules block critical traffic.

Best Practice: VPN Configuration

Use IKEv2 with AES-256 encryption for the IPsec tunnel. Enable Dead Peer Detection (DPD) to automatically re-establish the tunnel if one side goes offline. Document the Pre-Shared Key (PSK) in a secure password manager, never in a text file.

Phase 4: PowerShell User Provisioning

What Was Done

  • Wrote a PowerShell script to bulk-create user accounts in Entra ID from a CSV file, eliminating manual creation in the Admin Center.
  • Automated M365 license assignment (E3/E5) per user based on their department.
  • Automated Security Group membership assignment (e.g., "Sales" group gets SharePoint Sales Site access).
  • Added error handling and logging so failed provisioning attempts are logged to a CSV report file.

Sample PowerShell Script

# Connect to Microsoft Graph Connect-MgGraph -Scopes "User.ReadWrite.All" # Import users from CSV $users = Import-Csv -Path ".\new_users.csv" foreach ($user in $users) { try { # Create the user in Entra ID New-MgUser -DisplayName $user.DisplayName ` -UserPrincipalName $user.UPN ` -MailNickname $user.Alias ` -AccountEnabled ` -PasswordProfile @{ Password = $user.TempPassword ForceChangePasswordNextSignIn = $true } # Assign M365 License Set-MgUserLicense -UserId $user.UPN ` -AddLicenses @{SkuId = "your-sku-id"} Write-Host "✅ Created: $($user.DisplayName)" } catch { Write-Warning "❌ Failed: $($user.DisplayName) - $_" } }

Best Practice: Force Password Change

Always set ForceChangePasswordNextSignIn = $true when provisioning users. This ensures the temporary password is only used once, and the employee sets their own secure password on first login.

Precaution: Test with a Small Batch First

Never run a bulk provisioning script against all users on the first attempt. Test with 2-3 dummy accounts first, verify they appear correctly in Entra ID, then run the full batch. Always include try/catch error handling with logging.

Best Practice: Least-Privilege Permissions

When connecting to Microsoft Graph, request only the scopes you need (e.g., User.ReadWrite.All). Do not use Global Admin credentials for automation scripts. Create a dedicated service account with only the required permissions.

📋 Requirements for This Migration

To execute a migration like this for a real client, the following infrastructure and access requirements must be in place:

Microsoft 365 Tenant

An active M365 Business or Enterprise subscription with enough licenses for all users being migrated (E3 or E5 recommended).

Global Admin Access

Global Administrator credentials for the M365 tenant to configure Exchange Online, Entra ID, and manage licenses.

On-Premise AD Access

Domain Admin access to the on-premise Active Directory and Exchange Server for installing Azure AD Connect and running migration tools.

DNS Control

Access to the domain registrar's DNS management panel to update MX records, SPF, DKIM, and DMARC records during the email cutover.

Firewall & Switch Access

Administrative access to the edge firewall and core switches for VLAN creation, ACL configuration, and VPN tunnel setup.

Change Management Approval

Written approval from the client's stakeholders for the migration window, rollback plan, and expected downtime (if any).

⚠️ Pre-Migration Precaution Checklist

Before touching any production system, the following precautions must be taken to protect the client's data and business continuity:

  1. Full Backup: Export all mailboxes to PST files. Back up the Active Directory System State. Snapshot the Exchange Server VM (if virtual).
  2. Inventory Audit: Document every user account, distribution list, shared mailbox, and mail-enabled security group. Record current mailbox sizes.
  3. DNS TTL Reduction: 72 hours before migration, reduce MX and Autodiscover TTLs to 300 seconds to speed up DNS propagation during cutover.
  4. Stakeholder Communication: Send a company-wide email informing all employees of the migration date, expected duration, and who to contact if issues arise.
  5. Rollback Plan: Document the exact steps to revert MX records and re-enable on-premise Exchange if the migration fails. Keep the old server running for at least 2 weeks.
  6. Test Accounts: Migrate 2-3 pilot users first. Verify their email, calendar, and contacts transferred correctly before migrating everyone else.
  7. Conditional Access Policies: After migration, enable MFA (Multi-Factor Authentication) for all users and configure Conditional Access policies to block sign-ins from untrusted locations.
  8. Post-Migration Validation: After the cutover, send test emails to and from external addresses. Verify calendar sharing, distribution lists, and shared mailboxes all function correctly.
