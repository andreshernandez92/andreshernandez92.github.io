🛠️ Project Overview

I engineered this system to prove that enterprise-grade automation can be achieved without expensive SaaS subscriptions. This ERP runs entirely on Google Sheets as the database and Google Apps Script as the backend.

It fetches live product data from a free REST API, allows cashiers to build a cart through a responsive HTML Web App, generates a PDF receipt via the Google Drive API, and fires off a branded HTML email using GmailApp — all in under 3 seconds.

📸 Custom Google Sheets UI

The system injects a custom native menu into Google Sheets so non-technical staff can run reports and sync APIs without ever touching code.

🏗️ Advanced System Architecture

Unlike a standard monolithic spreadsheet which is prone to human error and data corruption, this ERP utilizes a robust Distributed Multi-Spreadsheet Architecture.

[INPUT LAYER: Web Apps] POS Sidebar.html | Expense WebApp.html | dummyjson.com REST API ↓ (Secure google.script.run) [APPS SCRIPT ENGINE: Backend] Concurrency control (LockService) | Rate Limiting (CacheService) Input Validation Engine | Config Reader ↓ (Automated Writing) [EXTERNAL DATA SHEETS: Isolated Databases] Smartbiz Orders | Smartbiz Inventory ↓ (Asynchronous Sync) [PROCESSING LAYER: Main Hub] P&L | Receipt PDF Generation | Live Dashboard | System Error Logs ↓ [OUTPUT LAYER: Automation] Gmail Alerts | Google Drive PDFs | Webhook JSON API Responses

Separation of Concerns (16 Modular Files)

To maintain enterprise code quality, the backend logic is split across 16 distinct .gs files. This prevents spaghetti code and makes the system easily extensible.

  • 01_Config.gs: Reads runtime settings so business limits (like tax rates) can be changed without coding.
  • 05_ApiIntegrations.gs: Handles UrlFetchApp calls to external endpoints.
  • 06_Validation.gs: A pure-function validation engine to sanitize all web app inputs.
  • 07_POS.gs: Manages the sales order pipeline and PDF receipt generation.
  • 13_WebApp.gs: Handles doGet() and doPost() routing for external webhooks.

🔒 Security & Concurrency (LockService)

When multiple cashiers submit orders simultaneously, standard Google Sheets scripts suffer from race conditions, resulting in overwritten rows. This project implements Google's LockService to ensure robust concurrency.

// Production-grade concurrency pattern used in the POS module: const lock = LockService.getScriptLock(); lock.waitLock(10000); // Wait up to 10 seconds for the queue try { // CRITICAL SECTION: Write to database, generate ID, export PDF } finally { lock.releaseLock(); }

Furthermore, the system employs Role-Based Access Control (RBAC). Cashiers only receive the URL to the POS Web App and cannot view the underlying Profit & Loss data. Managers have view-only access to specific tabs, while the Admin utilizes the Config Sheet to manage parameters securely.

⚡ Performance & Rate Limiting (CacheService)

The POS terminal fetches a live product catalogue from a public REST API. To prevent rate-limiting and ensure instant UI load times, the system utilizes CacheService.

// Caching pattern used in API Integrations module: const cache = CacheService.getScriptCache(); let raw = cache.get("product_catalogue"); if (!raw) { raw = UrlFetchApp.fetch("https://dummyjson.com/products").getContentText(); cache.put("product_catalogue", raw, 3600); // Cache for 1 hour }

📧 Automated Triggers & Webhooks

The system is entirely self-sustaining. It uses programmatically installed time-driven triggers to run background jobs:

  • Every 6 hours: Refreshes the product catalogue API.
  • Daily at 7:00 AM: Rebuilds the executive dashboard and P&L charts.
  • Daily at 8:00 AM: Scans inventory and sends Low-Stock email alerts.
  • Daily at 6:00 PM: Emails a beautifully formatted HTML End-of-Day Sales Report to the owner.

The script also exposes a doPost(e) webhook, meaning external tools like Zapier, Make.com, or Shopify can POST JSON data directly into the ERP database.
